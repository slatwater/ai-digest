# 微信小程序 BGM 与 WebGL Canvas 开发规则

**概要**：微信小程序的 `BackgroundAudioManager` 和 `<canvas type="webgl">` 存在若干平台级行为约束，必须在初始实现时就按以下规则配置，事后补救成本极高。

---

## 适用场景

- 小程序使用 `BackgroundAudioManager` 播放远程音频
- 小程序使用 `<canvas type="webgl">` 渲染视觉效果，并叠加 DOM 元素

---

## BGM 规则

### 规则 B1：远程音频必须在首屏初始化时预热 HTTP 缓存

`BackgroundAudioManager` 切换 `src` 有固定的 HTTP 下载 + decode + audio session 开销，大文件（>5MB）叠加 CDN 回源后真机首次延迟可达 1s+。

**做法**：在主页/首屏 `onLoad` 阶段，用 `InnerAudioContext` 静音播放**与 BGM 完全相同的 URL**：

```js
const ctx = wx.createInnerAudioContext();
ctx.volume = 0;         // 必须在 play() 前同步设置
ctx.src = resolvedUrl;  // 必须与 BGM 目标 URL 完全一致
ctx.play();             // 必须调用 play()，autoplay: false 不触发下载
ctx.onCanplay(() => {
  ctx.pause();
  setTimeout(() => ctx.destroy(), 100);
});
```

**约束**：
- 预热 `silence.mp3` 或其他占位 URL **无效**，仅能激活 audio session，不影响目标 URL 的 HTTP 缓存
- 若预热后 BGM 仍有延迟，说明当前平台两者缓存不共享，改用 `wx.cloud.downloadFile` 下载到 `tempFilePath` 再赋给 BGM

### 规则 B2：锁屏封面图只用 `cloud://` URL，不用本地路径

`coverImgUrl` 本地 `/assets/` 路径在真机当前微信版本被忽略，fallback 为默认图标。

### 规则 B3：iOS 锁屏亮屏后的首次音频卡顿是平台行为，不处理

每个场景首次亮屏时的短暂卡顿是 iOS Now Playing 卡片渲染的系统级行为，JS 层无法消除。**判断条件**：仅在首次 + 仅在锁屏亮屏后出现 → 直接跳过，不排查音频逻辑。

## WebGL Canvas 规则

### 规则 W1：所有主 Fragment Shader 必须使用 `precision highp float`

`mediump` 精度范围仅 ±16384：
- `sin(x) * 43000` 会溢出
- `pow(n, 30)`（n < 1）会被截为 0，导致依赖噪声的粒子效果整片消失

**规则**：所有主程序 fragment shader 首行写 `precision highp float;`。仅用于烘焙一次的 BG/lookup shader 可以例外。同时避免依赖 `pow` 高次方做稀疏峰，改用 `smoothstep` 阈值。

**注意**：`mediump` 在模拟器可能正常，真机必现，初始实现时就用 `highp`。

### 规则 W2：Canvas 必须用 `alpha: false`，背景在 Shader 内绘制

```js
const gl = canvas.getContext('webgl', { alpha: false });
```

原生组件 alpha 合成在真机不可靠，`alpha: true` 会导致黑屏。`alpha: false` 的连锁要求：
- Shader 内绘制全部内容（背景 + 效果），不依赖 DOM 透视
- 底图通过 `uBg` uniform 进 shader 采样

### 规则 W3：叠在 Canvas 上的 DOM 元素必须在 WXML 中声明于 `<canvas>` 之后

`<canvas type="webgl">` 是原生组件，CSS `z-index` 对其无效，层叠顺序由 WXML 声明顺序决定。

```xml
<!-- ✅ 正确 -->
<canvas type="webgl" id="gl-canvas"></canvas>
<view class="overlay-animation"></view>

<!-- ❌ 错误：overlay 会被 canvas 盖住，z-index 无法修复 -->
<view class="overlay-animation"></view>
<canvas type="webgl" id="gl-canvas"></canvas>
```

**排查信号**：DOM 动效数据正确、样式正确、屏幕上不可见 → 第一步检查 WXML 顺序。

### 规则 W4：必须用 DOM 遮罩 + 首帧事件驱动淡出来消除 Surface 黑闪

每次 `wx:if` 挂载 `<canvas type="webgl">`，原生层新建 Surface 到 JS 拿到 `gl` 对象之间有 **50–300ms 黑色暴露窗口**（视 Surface 热缓存命中情况而定，微信原生层约有 1–2 个热缓存槽）。CSS、`gl.clear`、shader 均无法覆盖此窗口。

**实现要点**：

1. WXML 中 canvas 同级添加遮罩，z-index 高于 canvas：
   ```xml
   <canvas type="webgl" id="gl-canvas"></canvas>
   <view class="surface-mask" style="opacity: {{maskGone ? 0 : 1}}"></view>
   ```

2. 遮罩初始 `opacity: 1`，背景色与 `gl.clearColor` 及 shader 背景起点**精确同色**（四层必须统一：CSS background / DOM mask / `gl.clearColor` / shader `mix` 第一参数）

3. **淡出触发必须是事件驱动**：在 render loop 的 `drawArrays` 执行后触发，加 `setTimeout(16)` 留一帧合成时间再 `setData({ maskGone: true })`，用 flag 保证只触发一次：
   ```js
   gl.drawArrays(...);
   if (!this._maskFadeStarted) {
     this._maskFadeStarted = true;
     setTimeout(() => this.setData({ maskGone: true }), 16);
   }
   ```

4. **不得使用固定 `setTimeout` 估算挡闪**：热缓存命中时 ~50ms 完成，冷启动时 200–300ms，固定值在两种情况下各有问题。

5. 加 500ms 兜底 setTimeout：防止 `_init` 失败（WebGL 不可用 / canvas node 找不到）导致遮罩永久显示。
